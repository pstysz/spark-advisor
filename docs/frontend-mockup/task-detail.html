Summary
The daily_etl_pipeline application has three interconnected performance problems forming a vicious cycle: data skew in Stage 4 forces one executor to handle 18x more data, which overwhelms its 2GB memory allocation, causing massive disk spill (12.4 GB) and triggering aggressive garbage collection (28% of runtime). Fixing the skew alone would likely resolve the spill and GC issues, but increasing executor memory provides a safety margin.
Causal Chain
Uneven key distribution in join (Stage 4) → Single partition receives 18x more data → Executor memory (2g) insufficient for skewed partition → Spill to disk (12.4 GB) → JVM GC pressure rises to 28% → Stage 4 wall time dominated by straggler task (245s vs 13s median) → Overall job duration inflated by ~35 minutes
Recommendations
1 Enable Adaptive Query Execution (AQE)
spark.sql.adaptive.enabled false → true

AQE will automatically detect and handle the data skew in Stage 4 by splitting the skewed partition into smaller sub-partitions. This addresses the root cause of the performance bottleneck.

Impact: ~70% total job duration reduction Risk: Low
2 Increase Executor Memory
spark.executor.memory 2g → 4g

Doubling executor memory will eliminate disk spill and reduce GC pressure. Even with AQE handling skew, additional memory provides headroom for data volume growth.

Impact: Eliminate 12.4 GB disk spill Risk: Medium — doubles memory cost
3 Optimize Shuffle Partitions
spark.sql.shuffle.partitions 200 → 148

With AQE enabled, you can also set spark.sql.adaptive.coalescePartitions.enabled=true to let Spark auto-tune this. Manual value of 148 targets 128MB per partition based on 18.6 GB shuffle volume.

Impact: Better task-level parallelism Risk: Low
